## [cite_start]Introduction [cite: 10]
[cite_start]In this lab, you will complete two tasks: [cite: 11]
1. [cite_start]Mount a Rowhammer attack on a real machine, taking the theory we learned in class to practice. [cite: 12]
2. [cite_start]Study what we can do to protect against Rowhammer. [cite: 13]

To begin, imagine DRAM as a matrix of cells. [cite_start]The core of Rowhammer is to access the cells that are adjacent to each other in a specific pattern. [cite: 14] [cite_start]In lecture, we studied how to design a reasonable access pattern to perform double-sided Rowhammer, and you may think the attack sounds easy to pull off. [cite: 15]

Unfortunately (or fortunately?), it is not. [cite_start]The most tricky part of the attack is dealing with address translations, and figuring out how exactly we can access the bit cells needed. [cite: 16] [cite_start]Note that all the addresses that a programmer can manipulate are **virtual addresses**, which need to be translated to **physical addresses**, and then to **DRAM cell coordinates**. [cite: 17]

[cite_start]**Virtual Address → Physical Address → DRAM Cell Coordinate** [cite: 18]

[cite_start]Since each step is not trivial, we have designed the lab to guide you to build your attack step by step, handling one of the translations at a time. [cite: 19] [cite_start]At the end of this lab, you will not only have a working Rowhammer attack, but will have also gained a deeper understanding of the complex addressing mechanism that is widely used in modern systems. [cite: 20]

### [cite_start]Linux pagemap interface [cite: 93]
[cite_start]Linux provides the `pagemap` interface that allows userspace programs to examine page tables. [cite: 96] [cite_start]For a running process, its pagemap can be accessed by reading the file `/proc/self/pagemap`. [cite: 97] [cite_start]This file is binary-encoded, containing a sequence of 64-bit values. [cite: 98] [cite_start]Each 64-bit value corresponds to a virtual page (assuming 4KB), where the N-th value corresponds to the N-th virtual page. [cite: 99]

* [cite_start]**Bit 63**: Indicates whether the page is present in memory. [cite: 108]
* [cite_start]**Bits 0-54**: If present, this field contains the **Physical Page Number (PPN)**. [cite: 109]

> [cite_start]**NOTE:** As a countermeasure for Rowhammer, `/proc/self/pagemap` has not been accessible by non-sudo users since Linux 4.0 (2015). [cite: 113] [cite_start]We provide a workaround through a kernel module. [cite: 115]

## [cite_start]Part 2: DRAM Geometry - Finding Bank Conflicts (15%) [cite: 180]
[cite_start]To perform Rowhammer on a specific victim row, you must locate attack rows by reverse engineering the physical address to DRAM coordinate mapping. [cite: 181] [cite_start]DRAM coordinates are typically a tuple: `<DIMM id, Channel id, Rank id, Bank id, Row id, Column id>`. [cite: 185]

For our lab machines:
* [cite_start]**Row id**: bits 17-31 [cite: 191]
* [cite_start]**Column id**: bits 0-12 [cite: 192]
* [cite_start]**Bank id**: 4-bit value (16 banks) derived by XORing some bits between 7 and 20. [cite: 189, 193]

## [cite_start]Part 5: Mitigation using ECC (20%) [cite: 300]
[cite_start]Explore Error Correcting Codes (ECC) as a defense. [cite: 301]

* [cite_start]**Repetition Codes**: Duplicates bits. [cite: 310]
* [cite_start]**Single Parity Bit**: XORs data bits to detect a single flip. [cite: 313]
* [cite_start]**Hamming Codes**: Uses multiple parity bits for SECDED (Single Error Correction, Double Error Detection). [cite: 318, 345]

### [cite_start]Hamming (22,16) [cite: 343]
[cite_start]This scheme uses 6 parity bits ($P0-P5$) to protect 16 data bits ($D0-D15$). [cite: 346]
[cite_start]$$P0 = D15 \oplus D13 \oplus D11 \oplus D10 \oplus D8 \oplus D6 \oplus D4 \oplus D3 \oplus D1 \oplus D0$$ [cite: 348]
(Additional equations for $P1-P4$ are provided in the manual)[cite_start]. [cite: 349, 350, 351, 352]
